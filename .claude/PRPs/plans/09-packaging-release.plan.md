# Plan 09: Packaging, Signing, Auto-Update & Release Pipeline

## Summary
Turn VibeMaestro from "clone + bun dev" into a real installable product. electron-builder configs for macOS (`.dmg`), Windows (`.exe`/NSIS), and Linux (`.AppImage`); macOS code signing + Apple notarization; Windows code signing (or honest SmartScreen warning); electron-updater wired against GitHub Releases for auto-update; semver `bun run release` script that bumps versions, tags, and pushes; GitHub Actions release workflow that builds + signs + publishes installers on tag.

## User Story
As a developer downloading VibeMaestro for the first time,
I want to download a signed installer and double-click it,
So that I'm running the app within 60 seconds — not cloning a repo and learning Bun.

## Problem → Solution
- **Current state (after plan #8):** `bun dev` opens the app on the implementer's machine. There's no `.dmg` to share, no auto-update path, no GitHub release, no installation experience for non-developers. Plans #1's `electron-builder install-app-deps` only handles dev-time native rebuilds — it does not build the actual installer.
- **Desired state:** `git tag v0.1.0 && git push origin v0.1.0` triggers a GitHub Actions workflow that builds signed `.dmg` (macOS, Apple notarized), `.exe` (Windows, signed if a cert is provided), and `.AppImage` (Linux), publishes them as a GitHub Release with auto-generated changelog, and notifies running v0.0.x clients via electron-updater that an update is available. Users on https://github.com/codeeatsleep2nd/VibeMaestro/releases see download links; clicking one installs the app in <60s.

### In-app update prompt

When electron-updater detects a newer release on GitHub:

```
┌────────────────────────────────────────────────────┐
│ ▎ Update available                                │
│   v0.2.0 is ready · changelog                     │
│   [ Restart and update ]    Later                 │
└────────────────────────────────────────────────────┘
```

- Shape: existing `info` toast variant (DESIGN.md §10)
- Position: bottom-right (toast stack)
- Auto-dismiss: never (this requires explicit user action)
- "Restart and update" calls `autoUpdater.quitAndInstall()`
- "Later" dismisses; reappears on next app launch (non-nagging cadence: max 1/day)

## Patterns to Establish

### RELEASE_TRIGGER_PATTERN — git tag IS the release event

```
Developer:           CI:                              Users:
  git tag v0.2.0     →  Build matrix runs           →  See update toast (electron-updater
  git push --tags          mac/win/linux                polls GitHub Releases hourly)
                          ↓
                       Sign + notarize (mac)
                          ↓
                       Publish GitHub Release
                       with .dmg, .exe, .AppImage
                       + latest-mac.yml/latest.yml/latest-linux.yml
```

No manual "press the release button." The tag is the event. The CHANGELOG.md (auto-generated from conventional commits since the last tag) becomes the release notes body.

### VERSION_BUMP_PATTERN — one command

```bash
bun run release:patch  # 0.1.0 → 0.1.1, generates changelog, tags
bun run release:minor  # 0.1.0 → 0.2.0
bun run release:major  # 0.1.0 → 1.0.0
```

Wraps `standard-version` (or equivalent). Always followed by `git push --follow-tags`.

### UNSIGNED_FALLBACK_PATTERN — honest scoping

If the project doesn't have signing certs:
- macOS build still produces `.dmg` (unsigned). README documents the right-click workaround.
- Windows build still produces `.exe` (unsigned). README documents SmartScreen workaround.
- Linux `.AppImage` doesn't need signing.
- electron-updater is configured but disabled when the platform's signing requirements aren't met (`autoUpdater.disableAutoUpdate = true`). README mentions: "Auto-update is enabled when signed builds ship; until then, check Releases page manually."

This is the plan-9 default. When/if certs are acquired, flip a feature flag.

## NOT Building

- **Windows EV cert.** Defer until there's a real Windows user base. v1 ships unsigned Windows with SmartScreen workaround documented.
- **Linux package-manager publishing.** No `apt`, `dnf`, `snap`, `flatpak`. `.AppImage` is the v1 distribution; package-manager listings can come later.
- **Mobile.** Out of scope for v1; v2 web mirror is a separate plan.
- **Delta updates.** electron-updater supports binary delta updates but requires server-side support beyond GitHub Releases. v1 uses full-installer updates.
- **Beta channel.** No `latest-beta.yml`; v1 has one channel (`latest`). When a beta channel is needed, electron-updater supports it via a config flag.
- **Crash reporting / telemetry.** No Sentry, no analytics. Worth a separate plan if/when there are users to report from.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Apple notarization fails | Medium | High (release blocked) | Build unsigned as fallback; document the manual workaround; signed-vs-unsigned is a CI env-var flag |
| GitHub Actions cross-platform matrix flake | Medium | Medium (release retry) | Pin runner versions; cache Bun + node-pty rebuilds; add `continue-on-error` for known-flaky platforms initially |
| electron-updater silently skips updates on unsigned builds | High (by design) | Medium (users stuck on old version) | Document explicitly in RELEASING.md and README; promote to signed when certs acquired |
| Native module ABI mismatch in packaged build | Medium | High (app crashes on launch) | Task 8's CI build-installer job catches this on every push to `main` |
| Cert renewal lapses | Low | High | RELEASING.md documents cert rotation; calendar reminder 30 days before expiry |
